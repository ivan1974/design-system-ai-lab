# Acceptance Checklist

## Purpose

Use this checklist to review screens generated by Figma Make with the
`design-system-ai-lab` package.

The goal is to confirm that the generated screen is:

- correctly connected to the published package
- composed with approved components, decision components and business patterns
- aligned with the four transversal principles
- informed by the knowledge layer when relevant
- visually aligned with the design system
- useful for the target user decision and documented user need
- evidence-aware and trustworthy
- ready for designer review and refinement

A generated screen does not need to be final.

It should be a strong first draft that respects the system rules.

---

## 1. Package setup

Check that the generated project uses the published npm package.

- [ ] The project depends on `design-system-ai-lab`.
- [ ] Components are imported from `design-system-ai-lab`.
- [ ] Styles are imported from `design-system-ai-lab/styles.css`.
- [ ] No local package named `design-system-ai-lab` was created.
- [ ] No `packages/design-system-ai-lab` folder was created.
- [ ] No local design system was generated to replace the package.
- [ ] No internal package paths are used.

Correct imports:

```tsx
import {
  ActionCard,
  ActionList,
  AlertCard,
  ConnectivityCoverageCard,
  CreateActionDialog,
  CustomerStatusCard,
  MetricCard,
  MetricGrid,
  PageHeader,
  PriorityList,
  RenewalRiskSummary,
  ValueProofCard,
} from "design-system-ai-lab";

import "design-system-ai-lab/styles.css";
```

Incorrect imports:

```tsx
import { Button } from "design-system-ai-lab/dist/components/button";
import { Button } from "./components/ui/button";
```

## 2. Forbidden local replacements

Check that Make did not recreate package components locally.

- [ ] No local `Button` replacement was created.
- [ ] No local `Card` replacement was created.
- [ ] No local `Badge` replacement was created.
- [ ] No local `Dialog` replacement was created.
- [ ] No local `MetricCard` replacement was created.
- [ ] No local `AlertCard` replacement was created.
- [ ] No local `ActionCard` replacement was created.
- [ ] No local `PageHeader` replacement was created.
- [ ] No local form components were created to replace `Field`, `Input`, `Select` or `Textarea`.
- [ ] No local decision components were created to replace `MetricGrid`, `PriorityList` or `ActionList`.
- [ ] No local business patterns were created to replace package patterns.
- [ ] No local wrappers were created for package components.
- [ ] No shadcn-like or Radix-based component library was generated locally.
- [ ] No MUI or unrelated component system was introduced.

Forbidden examples:

```tsx
import { Button } from "./components/ui/button";
import { Card } from "./components/ui/card";
```

```txt
packages/design-system-ai-lab/
src/components/ui/
```

## 3. Visible screen output

Check that the generated output is a real screen, not only setup code.

- [ ] `App.tsx` renders a complete visible screen.
- [ ] The screen is not empty.
- [ ] The screen is not only a component gallery.
- [ ] The screen is not only package setup.
- [ ] The screen has a clear page structure.
- [ ] The screen can be reviewed visually without additional implementation work.

If `App.tsx` is empty, use the correction prompt from `prompts/overview.md`.

---

## 4. Transversal principles

Check that the generated screen follows the four transversal principles.

- [ ] The screen follows `principles/accessibility.md`.
- [ ] The screen follows `principles/eco-design.md`.
- [ ] The screen follows `principles/ai-usage.md` when AI is present.
- [ ] The screen follows `principles/evidence-and-trust.md`.

The screen should not be accepted if it is:

- visually coherent but inaccessible
- visually polished but noisy
- AI-heavy without clear value
- unsupported by evidence
- disconnected from the user decision

---

## 5. Knowledge layer

Check that the generated screen reflects the knowledge layer when the screen
relates to known service experience needs, internal workflows or open validation
questions.

- [ ] Relevant knowledge-layer files were considered when applicable.
- [ ] The screen reflects relevant documented user needs when applicable.
- [ ] The screen reflects relevant design implications when applicable.
- [ ] The screen uses research-informed patterns when they match the screen intent.
- [ ] The screen does not treat open research questions as fully validated facts.
- [ ] Demo-ready assumptions are not presented as fully validated product standards.
- [ ] Research notes are not copied directly into the interface.
- [ ] Knowledge is translated into screen goal, pattern choice, evidence, actions or review criteria.
- [ ] `knowledge/asset-intelligence.md` is considered when the screen involves installed base, asset monitoring, connectivity, lifecycle, asset recommendations or asset-based value proof.
- [ ] Asset-heavy demo assumptions are not presented as complete, production-validated asset intelligence rules.

Relevant knowledge files:

```txt
knowledge/ux-insights.md
knowledge/user-needs.md
knowledge/design-implications.md
knowledge/tested-patterns.md
knowledge/open-questions.md
knowledge/asset-intelligence.md
```

Current evidence status:

```txt
Research-informed
Partially validated
Demo-ready
Not exhaustive
```

The screen should be revised if it looks polished but is disconnected from a
documented user need that applies to the prompt.

---

## 6. Screen goal

Check that the generated screen supports one clear user goal.

- [ ] The target user role is clear.
- [ ] The screen goal is clear.
- [ ] The relevant user need is clear when the prompt relates to a documented need.
- [ ] The screen pattern matches the prompt.
- [ ] The screen pattern matches the documented user need when applicable.
- [ ] The screen is not a generic dashboard.
- [ ] The screen helps the user answer what is happening.
- [ ] The screen helps the user answer why it matters.
- [ ] The screen helps the user answer what can be trusted.
- [ ] Asset-heavy screens help the user answer which assets are represented and what visibility is partial.
- [ ] The screen helps the user answer what to do next.

A strong generated screen should support decision-making, not just display data.

---

## 7. PageHeader

Check the screen orientation.

- [ ] The screen includes one main `PageHeader`.
- [ ] The `PageHeader` appears near the top of the screen.
- [ ] The title is specific.
- [ ] The description explains the user goal or decision context.
- [ ] The main action is explicit when present.
- [ ] The header is not generic, such as `Dashboard` or `Overview`.
- [ ] There are not multiple competing `PageHeader` components.

Good example:

```tsx
<PageHeader
  title="Customer monitoring"
  description="Understand customer status, trusted signals, risks and next actions."
  actions={<CreateActionDialog />}
/>
```

---

## 8. Business context and patterns

Check that business context is shown with the right pattern when possible.

- [ ] Business patterns are used when they match the section intent.
- [ ] `CustomerStatusCard` is used for customer status when relevant.
- [ ] `ConnectivityCoverageCard` is used for monitoring coverage when relevant.
- [ ] `RenewalRiskSummary` is used for renewal risk context when relevant.
- [ ] `ValueProofCard` is used for value proof when relevant.
- [ ] `StatusSummary` is used only when no more specific business pattern fits.
- [ ] Generic `Card` is not used to manually rebuild an existing business pattern.
- [ ] Context appears before detailed metrics, alerts or actions.
- [ ] Relevant trust, freshness, source scope or coverage limits are visible when they affect context.
- [ ] Asset scope, connectivity status, source strength and proof status are visible when they affect context.
- [ ] Business patterns are not replaced by generic cards when they match a research-informed need.
- [ ] There are not too many equal-importance context sections.

Good business pattern themes:

```txt
Customer status
Connectivity coverage
Renewal risk summary
Value proof
Generic status summary
```

---

## 9. Metrics

Check that metrics are decision-relevant.

- [ ] `MetricGrid` is used to group multiple metrics.
- [ ] `MetricCard` is used for metrics.
- [ ] Metrics support the screen goal.
- [ ] Metrics have clear labels.
- [ ] Metrics have interpretable values.
- [ ] Metrics include helper text when interpretation is needed.
- [ ] Metrics include freshness, validation or source context when trust depends on it.
- [ ] Trends are used only when useful.
- [ ] There are usually 2 to 4 metrics, not a large metric dashboard.
- [ ] Metrics are not vanity metrics.
- [ ] Metrics are not decorative.

Good metric examples:

```txt
Connected equipment
Open critical alerts
Overdue actions
Renewal readiness
Recommendations reviewed
Value proof status
Partially connected assets
Source strength
Expected outcome status
```

---

## 10. Alerts and risks

Check that alerts explain risk and next steps.

- [ ] `PriorityList` is used to group alerts, risks or blockers.
- [ ] `AlertCard` is used for risks, blockers or important signals.
- [ ] Every alert has a severity.
- [ ] Every alert has a specific title.
- [ ] Every alert has equipment or scope.
- [ ] Every alert has a meaningful description.
- [ ] Every alert includes a concrete recommendation.
- [ ] Critical alerts appear before lower-priority alerts.
- [ ] Severity matches the situation.
- [ ] Alerts are not generic notes.
- [ ] Important alert recommendations are supported by visible facts or context.
- [ ] Alerts do not hide partial visibility, missing proof or unresolved assumptions.
- [ ] Alert recommendations do not exceed the available evidence.
- [ ] Asset-related alerts include enough asset, source or connectivity context to make the recommendation reviewable.
- [ ] Asset-related alerts do not mix raw Health facts and Intelligence interpretation at the same visual level.

Good alert recommendation:

```txt
Plan a connectivity review with the customer and support team.
```

Bad alert recommendation:

```txt
Check.
```

---

## 11. Actions

Check that actions are concrete and assignable.

- [ ] `ActionList` is used to group recommended, planned or overdue actions.
- [ ] `ActionCard` is used for each action.
- [ ] Every action has a clear title.
- [ ] Every action has an owner.
- [ ] Every action has a due date.
- [ ] Every action has a priority.
- [ ] High-priority actions appear first.
- [ ] Action titles describe concrete work.
- [ ] Actions are connected to risks, metrics, recommendations or the user goal.
- [ ] Actions include enough context for handoff when another role must act.
- [ ] Actions are created for missing proof, missing ownership or unresolved gaps when appropriate.
- [ ] Asset-related actions include site, zone, room, asset group or asset context when needed for follow-through.
- [ ] Critical asset intelligence, modernization or value proof actions include human validation when needed.
- [ ] Actions are not vague notes.

Good action example:

```txt
Plan connectivity review with the customer
```

Bad action example:

```txt
Follow up
```

---

## 12. Forms and action creation

Check that forms use approved form components.

- [ ] `CreateActionDialog` is used for standard action creation flows.
- [ ] `Dialog` is used only for short focused flows not covered by a business pattern.
- [ ] Form fields use `Field`.
- [ ] Short values use `Input`.
- [ ] Limited choices use `Select`.
- [ ] Notes or longer text use `Textarea`.
- [ ] Form controls have visible labels.
- [ ] `htmlFor` and `id` are connected when possible.
- [ ] Placeholders are not used as labels.
- [ ] Raw inline-styled `input`, `select` or `textarea` elements are not generated.
- [ ] Disabled form controls are not used for display-only facts.

Good form pattern:

```tsx
<Field label="Owner" htmlFor="owner">
  <Input id="owner" placeholder="CSM" />
</Field>
```

---

## 13. Buttons

Check that buttons make actions explicit.

- [ ] `Button` is used for interactive actions.
- [ ] Button labels are explicit and action-oriented.
- [ ] There is one clear primary action per main section.
- [ ] Secondary actions use lower emphasis.
- [ ] Danger buttons are used only for destructive actions.
- [ ] Buttons are not used as status labels.
- [ ] Buttons are not used as badges.

Good labels:

```txt
Create action
Review risk
Plan follow-up
Prepare summary
Create mitigation action
```

Bad labels:

```txt
OK
Submit
More
Click here
```

---

## 14. Badges

Check that badges communicate concise status or metadata.

- [ ] `Badge` is used for short status, tone or metadata.
- [ ] Badge labels are short.
- [ ] Badge tones are valid.
- [ ] Badges are not used as buttons.
- [ ] Badges are not used for long explanations.
- [ ] `danger` badges are reserved for real critical states.
- [ ] Status is not communicated by color alone.
- [ ] Evidence-sensitive status includes readable context when needed.

Valid tones:

```txt
neutral
success
warning
danger
primary
```

Good badge:

```tsx
<Badge tone="warning">Source data requires review</Badge>
```

Weak badge:

```tsx
<Badge tone="warning">Warning</Badge>
```

## 15. Visual style

Check that the generated screen respects the visual direction.

- [ ] The screen feels sober.
- [ ] The screen feels B2B.
- [ ] The screen is readable.
- [ ] The screen is structured.
- [ ] The screen is evidence-aware.
- [ ] The screen avoids gradients.
- [ ] The screen avoids glassmorphism.
- [ ] The screen avoids decorative charts.
- [ ] The screen avoids arbitrary color palettes.
- [ ] The screen avoids custom visual systems.
- [ ] The screen avoids generic SaaS dashboard styling.
- [ ] Visual styling does not hide uncertainty.
- [ ] Unsupported recommendations are not styled as authoritative.

The visual style should support understanding, trust and actionability, not
decoration.

---

## 16. Token usage and local styling

Check that the generated screen respects the token system.

- [ ] The package CSS is imported once.
- [ ] Tokens are not redefined locally.
- [ ] No new color palette is introduced.
- [ ] No arbitrary radius scale is introduced.
- [ ] No arbitrary shadow system is introduced.
- [ ] Token-based utility classes are used when manual layout styling is needed.
- [ ] Package components are used before manual styling.
- [ ] Manual styling is not used to recreate existing components or patterns.
- [ ] Freshness, uncertainty or validation state is not hidden by styling.

Avoid local CSS like:

```css
:root {
  --ec-color-primary: #7c3aed;
}
```

## 17. Composition

Check that the screen follows a decision flow.

- [ ] The screen starts with orientation.
- [ ] Context appears before detailed signals.
- [ ] Visible facts appear before interpretation.
- [ ] Asset-heavy screens show asset context, connectivity, source scope and Health facts before Intelligence interpretation.
- [ ] Relevant knowledge-layer user needs are reflected when applicable.
- [ ] Trust, freshness, uncertainty or scope limits are visible when they affect the decision.
- [ ] Open research questions are not presented as fully validated facts.
- [ ] Metrics appear before or near risks when they support interpretation.
- [ ] Alerts explain what requires attention.
- [ ] Actions appear after the reason for action is clear.
- [ ] Expected outcomes, technical outcomes, internal proof and customer-ready proof are not collapsed into one status.
- [ ] The primary action is visible.
- [ ] The screen avoids random component order.
- [ ] The screen avoids large raw data tables.
- [ ] The screen avoids chart-first layouts unless explicitly requested.

Preferred flow:

```txt
PageHeader
→ Business context pattern
→ Trusted facts and key signals
→ MetricGrid with MetricCard items
→ PriorityList with AlertCard items
→ ActionList with ActionCard items
→ CreateActionDialog or Dialog when needed
```

For asset intelligence screens, use this additional flow:

```txt
Asset context
→ connectivity, source scope and source strength
→ raw Health or lifecycle facts
→ Intelligence interpretation
→ evidence-aware recommendation
→ owned or phased actions
→ expected outcome or value proof status
→ human validation when needed
```

## 18. Evidence and trust

Check that the screen makes trust reviewable.

- [ ] Observed facts are visible before interpreted signals.
- [ ] Recommendations are supported by visible or auditable evidence.
- [ ] Important facts include source, freshness or validation context when needed.
- [ ] Data uncertainty is visible when data may be incomplete, outdated or inferred.
- [ ] Partial visibility is visible when scope is limited.
- [ ] Source scope or coverage limits are shown when they affect the decision.
- [ ] Source strength is visible when it affects trust.
- [ ] Asset scope is visible when it affects trust.
- [ ] Proof gaps are visible when value proof is incomplete.
- [ ] AI outputs do not invent evidence.
- [ ] AI outputs are linked to visible or auditable facts when AI is used.
- [ ] Critical decisions include human validation.
- [ ] Display-only facts are not shown as disabled form fields.
- [ ] No fake citations are present.
- [ ] No fake sources are present.
- [ ] No invented proof points are present.
- [ ] No invented value outcomes are present.
- [ ] Open research questions are not presented as fully validated facts.
- [ ] Raw Health data is separated from Intelligence interpretation when both are present.
- [ ] Non-connected assets are not presented as live-monitored.
- [ ] Expected outcomes are not presented as proven value.
- [ ] Technical outcomes or internal proof are not presented as customer-ready proof without validation.
- [ ] Confidence language does not replace source evidence.
- [ ] Embedded components are not presented as top-level assets unless component-level investigation is required.

Recommended evidence flow:

```txt
Observed facts
→ interpreted signals
→ evidence-aware recommendations
→ owned actions
→ human validation when needed
```

Asset intelligence evidence flow:

```txt
Asset context
→ source-system Health facts
→ Intelligence interpretation
→ evidence-aware recommendation
→ owned or phased action
→ expected outcome or proof status
→ human validation when needed
```

## 19. AI usage

Check that AI is used only when it creates meaningful assistance.

- [ ] The screen is BI-first and AI-assisted when AI is relevant.
- [ ] Structured facts are displayed directly through components and patterns.
- [ ] GenAI is not used for basic retrieval.
- [ ] GenAI is not used for dates, owners, statuses, counts, lists, standard KPIs, asset coverage, monitoring freshness, renewal status, action ownership, asset hierarchy, telemetry values, connectivity status, lifecycle status or source scope.
- [ ] GenAI is used only for synthesis, explanation, prioritization, recommendation, drafting, reformulation, value proof synthesis, proof gap explanation or mitigation talking points.
- [ ] Visible facts appear before AI interpretation.
- [ ] AI recommendations are evidenced.
- [ ] AI wording does not hide partial visibility, missing proof or unresolved assumptions.
- [ ] AI-assisted value proof is grounded in visible or auditable facts.
- [ ] AI-assisted asset intelligence appears only after asset scope, source scope, connectivity and raw Health facts are visible.
- [ ] AI does not invent asset facts, telemetry, lifecycle status, source scope, expected outcomes or proven value.
- [ ] AI interpretation is not presented as source-system truth.
- [ ] Human review is visible for critical customer, contract, service, safety or compliance decisions.
- [ ] The screen is not prompt-first for basic structured data.

Bad AI usage:

```txt
Use a chatbot to retrieve the renewal date.
```

Good AI usage:

```txt
Use AI to prepare renewal talking points from visible renewal signals and proof gaps.
```

## 20. Content quality

Check that the content is useful and specific.

- [ ] The content is not generic filler.
- [ ] Customer or account context is concrete.
- [ ] Metrics have meaningful labels.
- [ ] Alerts explain impact.
- [ ] Recommendations are actionable.
- [ ] Recommendations are supported by visible facts.
- [ ] Recommendations do not exceed the available evidence.
- [ ] Value proof is grounded and not invented.
- [ ] Customer-ready proof is distinguished from internal-only or draft proof when relevant.
- [ ] Expected outcomes are labelled as expected until evidence proves them.
- [ ] Technical outcomes and internal proof are not worded as customer-ready proof without validation.
- [ ] Asset-heavy content states whether it is observed Health data, interpreted Intelligence, recommended action, expected outcome or proven value when relevant.
- [ ] Actions are assignable.
- [ ] Button labels describe outcomes.
- [ ] The screen avoids vague text such as `Data`, `Details`, `Issue`, `More` or `Follow up`.
- [ ] The screen avoids confident claims that are not supported by evidence.

Generated content should help the user decide, not just fill the layout.

## 21. Accessibility and clarity

Check for basic clarity and accessibility.

- [ ] Important statuses are expressed in text, not color alone.
- [ ] Interactive controls have explicit labels.
- [ ] Form fields have visible labels.
- [ ] Dialogs have meaningful titles.
- [ ] The layout is easy to scan.
- [ ] The reading order makes sense.
- [ ] Focus states have not been removed.
- [ ] The screen does not rely on decorative visuals to convey meaning.
- [ ] The screen does not rely on color alone to communicate critical information.
- [ ] Partial visibility, partially connected assets, non-connected assets and proof status are labelled in text.
- [ ] Users do not need to infer asset scope, source strength, Health versus Intelligence distinction or proof status from color, layout or AI wording alone.

---

## 22. Eco-design and density

Check that the generated screen is proportionate to the task.

- [ ] The screen is not over-generated.
- [ ] The number of sections is justified.
- [ ] Sections are justified by the user goal or documented user need.
- [ ] The number of metrics is limited and useful.
- [ ] Alerts are prioritized and limited.
- [ ] Actions are prioritized and limited.
- [ ] Duplicated content is avoided.
- [ ] Raw data dumps are avoided.
- [ ] Full asset dumps are avoided when grouping is enough for the decision.
- [ ] Compactness does not remove asset scope, connectivity status, source scope, source strength or proof status when these affect trust.
- [ ] Decorative assets are avoided unless useful.
- [ ] Interactions are necessary and task-driven.
- [ ] The generated code is readable and maintainable.

Recommended density:

```txt
1 PageHeader
1 to 2 business context patterns
2 to 4 MetricCard items inside MetricGrid
2 to 5 AlertCard items inside PriorityList
2 to 5 ActionCard items inside ActionList
0 to 1 CreateActionDialog or Dialog
```

## 23. Final acceptance

Accept the generated screen only if all critical checks pass.

Critical checks:

- [ ] `App.tsx` renders a complete visible screen.
- [ ] Components are imported from `design-system-ai-lab`.
- [ ] Styles are imported from `design-system-ai-lab/styles.css`.
- [ ] No local design system replacement was created.
- [ ] No local wrappers were created for package components.
- [ ] The screen has a clear user goal.
- [ ] The screen reflects relevant knowledge-layer user needs when applicable.
- [ ] Asset intelligence knowledge is reflected when the screen involves installed base, monitoring, lifecycle, connectivity, asset recommendations or asset-based value proof.
- [ ] The screen does not treat open research questions as fully validated facts.
- [ ] The screen uses approved components, decision components and business patterns.
- [ ] The screen follows accessibility principles.
- [ ] The screen follows eco-design principles.
- [ ] The screen follows AI usage principles when AI is present.
- [ ] The screen follows evidence and trust principles.
- [ ] Alerts include recommendations.
- [ ] Actions include owner, due date and priority.
- [ ] Important recommendations are supported by visible or auditable evidence and do not exceed that evidence.
- [ ] Asset-heavy screens do not hide asset scope, source scope, source strength, connectivity status or Health versus Intelligence distinction when these affect trust.
- [ ] Expected outcomes, technical outcomes, internal proof and customer-ready proof remain distinguishable when relevant.
- [ ] GenAI is not used for basic structured retrieval.
- [ ] No fake evidence, fake sources, fake citations, invented proof points or invented value outcomes are present.
- [ ] The screen remains sober and B2B.
- [ ] The screen helps the user decide what to do next.

---

## Final principle

The generated screen is acceptable when it uses the package correctly and helps
the user make a decision with enough clarity, evidence, trust and relevant
knowledge-layer grounding.

If it only looks polished but does not clarify the situation, risks, evidence,
trust, open assumptions, asset scope when relevant or next actions, it is not
acceptable yet.
