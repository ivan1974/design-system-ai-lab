# v0.8 Regeneration Plan

## Purpose

This guideline explains how to regenerate the design system after the v0.8 target surface reset.

The goal is not to keep improving the legacy surface. The goal is to regenerate the target Make Kit surface in a controlled order, then delete or archive the legacy surface.

## Source contract

Read first:

```txt
contracts/v0.8-target-surface.contract.json
contracts/v0.8-regeneration-plan.contract.json
```

## Strategy

```txt
do not polish legacy
regenerate target layers in order
keep tests transition-aware until regeneration completes
delete legacy after target surface is implemented
```

## Regeneration order

### 1. Foundations

Keep the prototype-derived visual system stable.

Source files:

```txt
src/design-system/foundations/tokens.css
src/design-system/styles.css
guidelines/tokens.md
guidelines/styles.md
guidelines/runtime/visual-rules.md
contracts/tokens.contract.json
contracts/visual-rules.contract.json
```

Do not create:

```txt
src/design-system/tokens.css
```

### 2. Core primitives

Regenerate:

```txt
Button
Card
Table
Tabs
Divider
Dialog
Popover
Tooltip
Checkbox
Field
Input
Label
Select
Textarea
```

Rules:

- use tokens from the canonical token file;
- do not create local visual grammar;
- do not preserve hardcoded prototype values outside tokens;
- regenerate Storybook coverage after the implementation stabilizes.

### 3. Semantic display

Regenerate only:

```txt
SemanticTag
SemanticPill
StatusIndicator
MetaLabel
```

Delete or internalize:

```txt
HealthPill
CoverageTag
DppTag
ConnectivityLabel
StatusWithIcon
PriorityPill
PriorityLabel
ProofStatus
SourceStrengthPill
StatusPill
```

Rules:

- use controlled tones;
- use contracts for mappings;
- do not create one semantic component per business value;
- use content and props for differences, not component proliferation.

### 4. Composition

Regenerate:

```txt
WorkspaceShell
MainNavigation
PageHeading
FilterBar
MasterDetailLayout
WorkspaceDetailPanel
SidePanel
SectionStack
SectionBlock
ListContainer
StickyActionBar
```

Rules:

- composition owns structure, not business meaning;
- panel behavior must be package-owned;
- shared transitional panel docs must be split or regenerated before release.

### 5. Decision blocks

Regenerate:

```txt
MetricBlock
DecisionBlock
EvidenceBlock
ActionBlock
StatusSummary
ReviewQueueRow
ActionRow
EvidenceRow
```

Delete or internalize:

```txt
MetricCard
AlertCard
RecommendationCard
ActionCard
EvidenceCard
InsightCard
RiskCard
CustomerCard
AssetCard
```

Rules:

- cards are not component types because content changes;
- reusable cognitive structure defines the component;
- business role differences belong in props, slots, mappings and contracts;
- move props from transitional cards to target blocks.

### 6. Screen patterns

Regenerate target patterns:

```txt
Installed Base Intelligence
Customer Monitoring
```

Installed Base remains the strict contractual screen.

Customer Monitoring becomes the second generation proof.

Application-specific wrappers must move to examples or be deleted.

### 7. Examples, stories, prompts and benchmarks

Regenerate:

```txt
guidelines/examples/golden/installed-base-intelligence.App.tsx
guidelines/examples/golden/customer-monitoring.App.tsx
guidelines/examples/golden/renewal-risk-review.App.tsx
guidelines/examples/golden/asset-recommendation-review.App.tsx
src/design-system/stories/
guidelines/prompts/
benchmarks/figma-make/cases/
benchmarks/figma-make/scoring/
```

Rules:

- all examples must share the same visual grammar;
- prompts must generate without local components;
- benchmarks must penalize card proliferation, local visual systems and generic dashboard drift.

## What not to do

Do not continue patching legacy components one by one unless they block target regeneration.

Do not preserve a public component because it already exists.

Do not create a new public component to express only a business label.

Do not update a golden example to use a transitional component unless the target component is not implemented yet and the contract marks it as transitional.

## Release switch

After regeneration, tests should become stricter again:

```txt
preferred components must be release-relevant in Storybook
preferred guidelines must use the full required reference structure
transitional props must move to target block props
legacy wrappers must be deleted or removed from public exports
```

## Exit criteria

The regeneration plan is ready when:

```txt
contracts/v0.8-regeneration-plan.contract.json exists
this guideline exists
v08 target surface tests reference the regeneration plan
runtime rules point only to target surface families
legacy is allowed only as a temporary bridge
```
