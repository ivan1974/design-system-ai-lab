# Workspace visual v4

## Purpose

This file defines the visual composition rules for v0.4 workspaces generated by Figma Make.

It complements `workspace-v2.md`, `visual-brand.md`, `tokens.md` and `styles.md`.

---

## Required workspace shape

A v0.4 workspace should usually follow this order:

```txt
PageHeading
FilterBar or SecondaryNavigation
MasterDetailLayout
ListContainer with QueueRows
WorkspaceDetailPanel
Tabs inside detail where useful
StickyActionBar or ActionRow follow-through
```

Use this shape when the user reviews customers, assets, risks, recommendations or QBR readiness.

---

## Central workspace preservation

When detail is optional or competes with the central workspace, use:

```tsx
<MasterDetailLayout detailOpen={detailOpen} detailMode="overlay" />
<WorkspaceDetailPanel open={detailOpen} mode="overlay" />
```

Use inline detail when the selected item should remain permanently visible.

Use overlay detail when the user needs the list or central workspace to keep full width.

Do not create a custom drawer or side panel.

---

## Row-first lists

Use `ListContainer` with queue rows for repeated review objects.

Preferred:

```tsx
<ListContainer>
  <AssetQueueRow ... />
  <AssetQueueRow ... />
</ListContainer>
```

Avoid:

```tsx
<div className="rounded-lg border bg-white">
  <div className="p-4 hover:bg-gray-50">...</div>
</div>
```

---

## Detail panel content

Use `WorkspaceDetailPanel` for selected-item review.

Inside the panel, prefer:

```txt
Tabs
SectionStack
SectionBlock
EvidenceRow
RecommendationReviewPanel
RecommendationCard
AlertCard
ValueProofCard
AssetIntelligenceSummary
```

The panel should answer:

```txt
What is selected?
What is known?
What is uncertain?
What source supports it?
What should the user do next?
```

---

## Density rules

A v0.4 workspace should be dense enough for operational work.

Use:

- compact metrics for signals
- rows for repeated objects
- business patterns for known sections
- one sticky or primary action area

Avoid:

- one card per fact
- one card per action
- large decorative hero sections
- excessive whitespace
- hidden source context

---

## Visual acceptance criteria

A v0.4 workspace passes if:

- page hierarchy starts with `PageHeading`
- major sections use `SectionHeading` or `SectionBlock`
- repeated objects use queue rows
- queue rows live inside `ListContainer`
- selected item detail uses `WorkspaceDetailPanel`
- panel detail can be inline or overlay
- tabs use `Tabs` or `HeaderTabs`
- surfaces remain white-first
- no local drawer, tab, card, chip or row system is created
- decision-critical source context stays visible
