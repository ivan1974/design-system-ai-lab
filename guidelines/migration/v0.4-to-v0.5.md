# Migration Guide — v0.4 to v0.5

## Purpose

Use this guide to migrate existing Make guidance, generated screens and package usage to v0.5.0.

v0.5.0 introduces the Controlled Generation Standard.

The release is not only a visual or component update. It adds testable rules for Figma Make generation.

---

## What changed

### 1. New contracts folder

v0.5.0 adds machine-readable contracts:

```txt
contracts/package.contract.json
contracts/components.contract.json
contracts/props.contract.json
contracts/screen-architecture.contract.json
contracts/business-patterns.contract.json
contracts/evidence-and-trust.contract.json
contracts/generation-blockers.contract.json
```

Use these for tests, automated checks and future tooling.

Do not treat them as a second source of truth. They are the testable form of active guidelines.

---

### 2. Stricter critical props

Critical fields are now controlled more strictly.

This affects dimensions such as:

```txt
source strength
validation status
proof readiness
customer readiness
human validation
action priority
```

Do not invent string values for trust, evidence, validation, readiness or action-critical fields.

Use documented values from:

```txt
contracts/props.contract.json
src/design-system/types/
```

---

### 3. Updated Make entry point

Figma Make should now read:

```txt
guidelines/make-minimal-contract.md
```

before full guidance.

Recommended order:

```txt
guidelines/make-minimal-contract.md
→ guidelines/Guidelines.md
→ guidelines/setup.md
→ specific guideline files when needed
→ contracts/ for machine-readable checks
```

`Guidelines.md` and `setup.md` remain active entry points.

The minimal contract only makes the blocking rules easier for Make to prioritize.

---

### 4. New benchmark process

v0.5.0 adds a Figma Make benchmark in:

```txt
benchmarks/figma-make/
```

Use it to test real Make generations.

Benchmark cases live in:

```txt
benchmarks/figma-make/cases/
```

Scoring lives in:

```txt
benchmarks/figma-make/scoring/scoring-model.md
benchmarks/figma-make/scoring/scoring-template.md
```

Use benchmark outputs to compare normal and adversarial prompts.

---

### 5. Golden examples as tested fixtures

Golden examples remain in:

```txt
guidelines/examples/golden/
```

They are now used as tested compliance fixtures.

Current examples:

```txt
customer-monitoring.App.tsx
renewal-risk-review.App.tsx
asset-recommendation-review.App.tsx
qbr-readiness.App.tsx
installed-base-explorer.App.tsx
```

Run:

```bash
npm run test:generation
```

---

### 6. Deprecated free-text statuses

Free-text status values are deprecated for trust-critical fields.

Avoid values such as:

```txt
strong-ish
AI validated
probably ready
customer proof available
high confidence
fully monitored
```

Use controlled values instead.

Examples:

```txt
SourceStrength
ValidationStatus
ProofReadiness
HumanValidationRequirement
ActionPriority
```

## Required migration actions

### For generated screens

Check that screens:

- import only from `design-system-ai-lab`
- import `design-system-ai-lab/styles.css`
- do not create local design-system components
- use decision workspace structure for list/detail review
- use business patterns before manual reconstruction
- show facts before interpretation
- do not invent evidence, source facts, telemetry or value proof
- keep human validation visible for sensitive decisions
- use `ActionRow` or `ActionCard` with owner, due date and priority

---

### For prompts

Update prompts to reference:

```txt
guidelines/make-minimal-contract.md
guidelines/Guidelines.md
guidelines/setup.md
```

Prompts should tell Make to:

- compose from the package root
- use the stylesheet entry point
- create one decision workspace
- use business patterns first
- show facts before interpretation
- avoid invented evidence
- keep action ownership visible

---

### For review

Use the review loop:

```txt
guidelines/evaluation/review/review-loop.md
```

Use repair routing:

```txt
guidelines/evaluation/review/repair-routing.md
```

If a generated screen fails blocking review, repair before quality review.

---

## Breaking changes to watch

- Action components without owner, due date or priority should fail.
- Alert components without recommendation should fail.
- Free-text proof, source or validation statuses should be replaced.
- Local component wrappers should be removed.
- Generic dashboard/card-stack outputs should be repaired into decision workspaces.
- Expected outcomes must not be presented as proven value.

---
