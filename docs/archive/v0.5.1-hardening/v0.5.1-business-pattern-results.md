## Pattern ownership rules

### Customer and review context

```txt
CustomerStatusCard → customer-level orientation
CustomerReviewReadinessCard → readiness for customer-facing review
```

Use these patterns before rebuilding customer context with `Card`, `StatusSummary` or `KeyValueList`.

---

### Coverage and risk context

```txt
ConnectivityCoverageCard → monitoring coverage and visibility limits
ServiceRiskSummary → service risk orientation before detailed review
```

Coverage is not proof.

Service risk summary is not an action plan.

---

### Action creation

```txt
CreateActionDialog → standard action creation flow
```

The source implementation was also aligned so priority includes:

```txt
critical
high
medium
low
```

---

### Structural hierarchy

```txt
ComponentHierarchy → asset or component structure
```

Hierarchy is structure only.

It is not proof, recommendation or action ownership.

---

### Queue rows

```txt
AssetQueueRow
CustomerQueueRow
RecommendationQueueRow
RiskQueueRow
```

These rows are covered by:

```txt
guidelines/patterns/queue-row-patterns.md
```

Reason:

```txt
The four queue row components share the same business pattern: dense triage and selection. A shared guideline avoids duplication and keeps the decision rule consistent.
```

Queue rows are alternatives to card stacks.

They should be used inside `ListContainer` or queue sections when the user must scan repeated business objects.
