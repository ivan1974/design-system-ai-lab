# v0.5.1 Controlled Values Alignment

## Purpose

Sprint 4 aligns controlled vocabulary across source types, contracts and domain-model guidelines.

Goal:

```txt
Make controlled vocabulary impossible to misinterpret.
```

A generative AI must not invent values for trust, proof, readiness, priority, status, severity, tone, mode, density or risk.

## Canonical values

### Source strength

```txt
unknown
partial
single-source
multi-source
validated
```

Deprecated aliases remain accepted only for compatibility:

```txt
strong
needs_review
internal
customer_ready
```

New documentation and generated screens must use canonical values.

---

### Validation status

```txt
not-reviewed
internal-review-needed
internally-validated
customer-ready
blocked
```

Use `validationStatus` for review state.

---

### Customer readiness

```txt
internal
needs-review
customer-ready
```

Deprecated aliases remain accepted only for compatibility:

```txt
needs_review
customer_ready
```

Use hyphenated values in new documentation and generated screens.

---

### Human validation

```txt
not-required
recommended
required
```

Use `required` when a sensitive recommendation, proof claim or customer-facing action requires human review.

---

### Proof readiness

```txt
not-available
expected-only
internal-proof
customer-ready-proof
```

Use `expected-only` for projected or illustrative outcomes.

Use `internal-proof` when proof exists but is not ready for customer use.

Use `customer-ready-proof` only when validation and customer use are explicit.

---

### Evidence source type

```txt
source-system
expert-review
customer-input
ai-generated
unknown
```

Use `ai-generated` only for generated synthesis, interpretation or drafting.

Do not use generated content as source-system fact.

---

### Action priority

```txt
low
medium
high
critical
```

---

### Action status

```txt
todo
in-progress
blocked
done
```

Deprecated alias accepted only for compatibility:

```txt
in_progress
```

Use `in-progress` in new documentation and generated screens.

Do not use action completion as value proof.

---

### Risk level

```txt
critical
warning
info
```

Use risk level only for risk, alert or service-risk interpretation.

Do not use it as action priority or proof readiness.
