# v0.5.0 success metrics

## Purpose

This file defines how to evaluate whether `v0.5.0` successfully moves Design System AI Lab from documented AI governance to controlled generation reliability.

The metrics are intentionally simple.

They should be understandable by designers, product owners and developers.

They should also be concrete enough to guide tests, benchmark reviews and release decisions.

---

## Measurement principle

A generated screen should not be accepted only because it looks coherent.

For v0.5.0, a generated screen should be evaluated against six dimensions:

```txt
1. Package contract
2. Workspace architecture
3. Component and business pattern usage
4. Evidence and trust integrity
5. Actionability
6. Visual sobriety and accessibility basics
```

These dimensions produce a generation compliance score.

---

## Headline targets

`v0.5.0` should meet these release targets:

```txt
Generation compliance score >= 80%
Blocking failure rate <= 20%
No local design system recreation in accepted benchmark outputs
No invented evidence in accepted benchmark outputs
All golden examples pass automated generation checks
All critical components have documented contract coverage
```

---

## Metric 1 — Generation compliance score

### Target

```txt
>= 80%
```

### What it measures

The average quality and rule compliance of benchmark outputs.

### Scoring model

Use a 100-point score:

| Dimension | Points |
| --- | ---: |
| Package contract | 20 |
| Workspace architecture | 20 |
| Component and business pattern usage | 15 |
| Evidence and trust integrity | 20 |
| Actionability | 15 |
| Visual sobriety and accessibility basics | 10 |
| Total | 100 |

### Interpretation

| Score | Status | Meaning |
| ---: | --- | --- |
| 90-100 | Accepted | Strong controlled generation output |
| 80-89 | Accepted with minor improvements | Valid first draft |
| 65-79 | Needs repair | Not acceptable before repair |
| < 65 | Rejected | Invalid generation |

### Release criterion

The average score across benchmark prompts should be at least 80.

No accepted output should contain a blocking trust or package failure.

---

## Metric 2 — Blocking failure rate

### Target

```txt
<= 20%
```

### What it measures

The percentage of benchmark outputs that fail at least one blocking review item before repair.

### Blocking failures include

```txt
missing visible App.tsx screen
missing package stylesheet import
forbidden internal imports
local design system recreation
local visual component duplicates
missing workspace structure for review flows
card stack drift for decision workflows
invented evidence or proof
expected outcomes presented as proven value
actions without owner, due date or priority
hidden human validation for critical decisions
```

### Calculation

```txt
blocking failure rate = outputs with at least one blocking failure / total benchmark outputs
```

### Release criterion

The benchmark should show a blocking failure rate of 20% or less after v0.5 guidance, contracts and tests are in place.

---

## Metric 3 — Local design system recreation

### Target

```txt
0 accepted benchmark outputs
```

### What it measures

Whether Figma Make recreates local design system components instead of using the package.

### Fail examples

```txt
local Button
local Card
local Badge
local Tabs
local Drawer
local SidePanel
local Pill
local Metric
local QueueRow
local form primitives
local components/ui folder
local packages/design-system-ai-lab folder
```

### Release criterion

No accepted benchmark output should recreate local design system components or folders.

If a generated output does this, it must be rejected or repaired before acceptance.

---

## Metric 4 — Invented evidence prevention

### Target

```txt
0 accepted benchmark outputs
```

### What it measures

Whether generated screens invent sources, telemetry, asset facts, proof, validation or value evidence.

### Fail examples

```txt
invented source presented as real
invented evidence presented as real
invented telemetry presented as source-system fact
expected saving presented as proven value
AI confidence presented as source strength
partial asset visibility presented as complete monitoring
internal proof presented as customer-ready proof without validation
```

### Release criterion

No accepted benchmark output should invent or overstate evidence.

Outputs that include hypothetical, expected or illustrative values must label them clearly as such.

---

## Metric 5 — Golden example automated checks

### Target

```txt
100% pass
```

### What it measures

Whether golden examples comply with the package contract and generation rules.

### Minimum checks

```txt
imports only from design-system-ai-lab
imports design-system-ai-lab/styles.css
no forbidden internal imports
no local visual component duplicates
uses workspace structure where relevant
uses row-first patterns for repeated objects
uses business patterns when relevant
keeps evidence and validation context visible
keeps follow-through actions explicit
```

### Release criterion

All golden examples must pass automated generation checks before v0.5.0 release.

---

## Metric 6 — Critical component contract coverage

### Target

```txt
100% of critical components documented in contracts
```

### Critical components

Initial critical list:

```txt
ActionCard
ActionRow
AlertCard
RecommendationCard
RecommendationReviewPanel
AssetIntelligenceSummary
ValueProofCard
RenewalRiskSummary
ServiceRiskSummary
SourceStrengthPill
StatusPill
PriorityPill
```

### What coverage means

A critical component has contract coverage when the repository documents:

```txt
allowed enum-like prop values
required trust or action props where relevant
forbidden misuse patterns where relevant
example usage in a golden example or Storybook story
related review criteria
```

### Release criterion

Every critical component must be represented in the relevant contract file or explicitly marked as deferred with a reason.

---

## Metric 7 — Repair effectiveness

### Target

```txt
>= 70% of repairable benchmark failures improve after routed repair
```

### What it measures

Whether repair prompts and review routing help Make produce better outputs after a failed generation.

### Calculation

```txt
repair effectiveness = repaired outputs with improved score / repairable failed outputs
```

### Release criterion

At least three benchmark outputs should document:

```txt
initial failure
selected repair prompt
revised output score
final decision
```

This metric is a v0.5 maturity signal.

It should not block early implementation work, but it should be measured before release.

---

## Metric 8 — Scope control

### Target

```txt
No major out-of-scope expansion in v0.5.0
```

### What it measures

Whether v0.5.0 remains focused on controlled generation reliability.

### Watchouts

Avoid turning the release into:

```txt
a broad visual redesign
a large component expansion
a multi-brand theming project
a full production-hardening project
a runtime analytics project
```

### Release criterion

Most v0.5.0 changes should support at least one of these outcomes:

```txt
more explicit contracts
more reliable tests
more constrained critical components
stronger golden examples
clearer benchmark evaluation
better repair routing
clearer Make usage
```

## Release decision rule

`v0.5.0` should not be released because every possible generation is perfect.

It should be released when the project can show that generation quality is:

```txt
framed by explicit scope
constrained by contracts
checked by tests
anchored in golden examples
measured through benchmark scoring
repairable through documented prompts
```

The target is not perfection.

The target is controlled, reviewable and measurable first-draft generation.
