# v0.5.0 roadmap — Controlled generation standard

## Purpose

This roadmap defines the scope, success criteria and sprint plan for `v0.5.0`.

`v0.5.0` is a hardening release.

Its purpose is to move Design System AI Lab from a documented AI governance prototype to a more reliable controlled generation system for Figma Make.

The goal is not to expand the visual library.

The goal is to make AI-assisted generation more constrained, testable, repairable and measurable.

---

## Documentation cleanup principle

Obsolete roadmap and transition documents should not stay in the active `guidelines/` documentation tree.

Figma Make needs a small number of current, non-conflicting sources of truth.

Historical v0.4 transition files were removed from the active documentation to avoid guidance debt and stale references.

For v0.5.0, active planning belongs in:

```txt
docs/roadmap/v0.5.0-roadmap.md
docs/roadmap/v0.5.0-success-metrics.md
```

Active generation guidance should remain in `guidelines/` only when it is current and intended for Make or reviewers.

---

## Vision v0.5.0

`v0.5.0` turns the project into a controlled generation standard.

A valid Figma Make generation should no longer be accepted only because it looks plausible or uses some design system components.

It should be checked against:

```txt
package usage
→ documented component contracts
→ workspace architecture
→ business pattern selection
→ evidence integrity
→ actionability
→ human validation visibility
→ measurable generation quality
```

The design system should continue to help AI compose useful first drafts.

But the system should now make invalid generations easier to detect, reject, repair and measure.

---

## Release positioning

`v0.5.0` is:

```txt
a controlled generation reliability release
```

It is not:

```txt
a visual redesign release
an expansion release
a production-readiness release
```

The main product shift is:

```txt
From: documented governance for AI-assisted interface generation
To: testable and measurable controlled generation for decision workspaces
```

## Problem to solve

The current project already defines strong guidance for Figma Make:

- import from the package root
- use the package stylesheet
- avoid local design system recreation
- prefer decision workspaces over generic dashboards
- use rows and compact primitives for repeated operational information
- use business patterns when available
- show facts before AI interpretation
- avoid invented evidence or value proof
- keep human validation visible for critical decisions

However, too many of these rules are still primarily expressed as Markdown guidance.

For v0.5.0, the project should make the most important rules:

```txt
explicit
machine-readable where useful
testable through generation-rule tests
visible in golden examples
measurable through prompt benchmark scoring
repairable through routed repair prompts
```

---

## In scope

The following work is in scope for `v0.5.0`.

### Controlled generation contracts

Create a small set of machine-readable contracts for the most important generation rules.

Initial contract areas:

```txt
package import contract
component and prop contract
screen architecture contract
business pattern contract
evidence and trust contract
generation blocker contract
```

The contracts should not become a second design system documentation site.

They should capture only the rules that need to be checked or reused by tests.

### Generation-rule tests

Add automated tests that check golden examples and future generated outputs against the most important rules.

Priority checks:

```txt
allowed package imports
required stylesheet import
forbidden internal imports
forbidden local design system recreation
required workspace structure when relevant
row-first repeated object structure
business pattern usage when relevant
action ownership and follow-through
evidence and proof integrity
```

### Tested golden examples

Golden examples should become compliance references, not only visual references.

They should be testable against the contracts and generation rules.

### Critical prop hardening

Critical components should carry more of the governance in TypeScript.

Priority dimensions:

```txt
source strength
validation status
proof readiness
customer readiness
action priority
action owner
action due date
human validation requirement
```

This should focus on trust-sensitive and action-sensitive components first.

### Make minimal contract

Create a short, operational entry-point file for Figma Make.

It should summarize the non-negotiable rules without duplicating all detailed guidelines.

### Prompt benchmark

Create a small benchmark of normal and adversarial Figma Make prompts.

The benchmark should test whether Make follows the package contract, workspace structure, evidence rules and actionability rules.

### Structured repair loop

Connect blocking failures to repair prompts.

The desired loop is:

```txt
prompt
→ generation
→ blocking review
→ score
→ repair prompt
→ revised generation
→ final score
```

### Release documentation

Document what changed in v0.5.0 and how it differs from the previous prototype stage.

---

## Out of scope

The following work is intentionally out of scope for `v0.5.0`.

```txt
complete visual redesign
massive addition of new components
full production readiness
advanced CI/CD publication automation
multi-brand support
dark mode theming
runtime analytics
large-scale documentation rewrite
retaining obsolete roadmap documents in active guidelines
```

These items can be reconsidered after the controlled generation foundation is stable.

## Success metrics

The detailed success metrics are defined in:

```txt
docs/roadmap/v0.5.0-success-metrics.md
```

The headline targets are:

```txt
Generation compliance score >= 80%
Blocking failure rate <= 20%
No local design system recreation in accepted benchmark outputs
No invented evidence in accepted benchmark outputs
All golden examples pass automated generation checks
All critical components have documented contract coverage
```

## Sprint plan

### Sprint 0 — v0.5.0 framing and cleanup

Objective:

Define what v0.5.0 must solve, what it must not solve, how success will be measured, and which obsolete planning documents should be removed from active guidance.

Deliverables:

```txt
docs/roadmap/v0.5.0-roadmap.md
docs/roadmap/v0.5.0-success-metrics.md
obsolete v0.4 transition files removed from active guidelines
stale references removed
```

Acceptance criteria:

```txt
v0.5.0 scope is documented
success metrics are explicit
release is positioned as a reliability release, not a library expansion
obsolete v0.4 roadmap files are not left in active guidance
```

---

### Sprint 1 — Repository and documentation foundation

Objective:

Stabilize the repository foundation before adding controlled generation logic.

Expected outputs:

```txt
fixed markdown lint paths
standard check script
verified package build
verified Storybook build
clear Radix bundling or externalization decision
```

Acceptance criteria:

```txt
npm run lint:md targets the current documentation folders
npm run check exists
package build passes
Storybook build passes
package import and stylesheet contract remain aligned with the guidelines
```

---

### Sprint 2 — Machine-readable generation contracts

Objective:

Create the first contract layer used by tests and future generation checks.

Expected outputs:

```txt
contracts/package.contract.json
contracts/components.contract.json
contracts/props.contract.json
contracts/screen-architecture.contract.json
contracts/business-patterns.contract.json
contracts/evidence-and-trust.contract.json
contracts/generation-blockers.contract.json
```

Acceptance criteria:

```txt
contracts are valid JSON
contracts do not duplicate full documentation
contracts map to existing guidelines and review rules
contracts can be consumed by tests
```

---

### Sprint 3 — Critical component hardening

Objective:

Move the most important trust and action rules from documentation-only guidance into TypeScript contracts.

Expected outputs:

```txt
shared trust, evidence and action types
stricter props for critical components
updated stories
updated golden examples
updated guidelines where prop values change
```

Acceptance criteria:

```txt
critical action components require owner, due date and priority where appropriate
critical alert components require a recommendation where appropriate
trust-sensitive props use documented controlled values
old free-text patterns are still supported only where intentional
```

---

### Sprint 4 — Golden examples as tested fixtures

Objective:

Turn golden examples into tested compliance references.

Expected outputs:

```txt
generation-rule tests for golden examples
import contract checks
workspace structure checks
business pattern checks
evidence integrity checks
actionability checks
```

Acceptance criteria:

```txt
all golden examples pass generation-rule tests
tests fail on forbidden imports
tests fail on local design system recreation
tests fail when required workspace structure is missing
tests fail when critical actions lack ownership or priority
```

---

### Sprint 5 — Figma Make prompt benchmark

Objective:

Measure how reliably the system guides Figma Make.

Expected outputs:

```txt
benchmarks/figma-make/prompts/
benchmarks/figma-make/evaluations/
benchmarks/figma-make/scoring/
normal prompts
adversarial prompts
scoring model
```

Acceptance criteria:

```txt
benchmark includes core decision workspace prompts
benchmark includes adversarial prompts
outputs can be scored consistently
benchmark scoring feeds the v0.5.0 success metrics
```

---

Make generation failures easier to diagnose and repair.

```txt
each blocking failure has a recommended repair path
repair prompts are linked to review criteria
at least three benchmark outputs document before and after repair scores
```

---

## Governance principle

Do not use v0.5.0 to make the system bigger before making it more reliable.

For this release, prefer:

```txt
fewer new components
more explicit contracts
more test coverage
clearer examples
better repair loops
measurable generation quality
less stale guidance
```

The design system should become harder for AI to misuse before it becomes broader.
