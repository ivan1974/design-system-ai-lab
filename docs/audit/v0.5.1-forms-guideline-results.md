## Core decisions

### Forms must support a clear user task

Generated forms are allowed only when the user can submit, save, confirm, review or create something.

Forms should not appear as random editable areas inside a decision workspace.

---

### Field is the default wrapper

Use:

```txt
Field + Input
Field + Select
Field + Textarea
```

`Field` provides the label, helper and error structure.

For most generated forms, standalone `Label` should not be needed.

---

### Label relationships are mandatory

When a form control has an `id`, `Field.htmlFor` or `Label.htmlFor` should match it.

Required relationship:

```txt
Field.htmlFor → Input.id
Field.htmlFor → Select.id
Field.htmlFor → Textarea.id
```

Placeholder text is supplemental guidance, not the label.

---

### Raw controls are forbidden when package controls fit

Do not generate raw controls when package components exist:

```txt
<input>
<select>
<textarea>
<label>
```

Use package components:

```txt
Input
Select
Textarea
Label
Field
```

---

### Select owns controlled choices

Use `Select` when the value must match controlled vocabulary.

Examples:

```txt
priority
status
sourceStrength
proofReadiness
validationStatus
```

Do not use free text for controlled values.

---

### CreateActionDialog owns standard action creation

For action creation flows, prefer:

```txt
CreateActionDialog
```

Use raw form composition only for short custom forms not covered by the action dialog pattern.

Action-related forms must preserve:

```txt
owner
due date
priority
```
