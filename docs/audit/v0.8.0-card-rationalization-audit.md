# v0.8.0 Card Rationalization Audit

## Status

Sprint 1 audit for v0.8.0.

This document audits card-like components across `components`, `decision` and `patterns`. It identifies which cards should remain as core DS structures and which should be merged, internalized or deleted.

## Core finding

The design system already has a usable `Card` primitive, but multiple public components recreate similar card structures with business-specific names.

The repeated structure is:

```txt
surface / border / radius / padding
  header or badge row
  title and description
  evidence / metadata / key-value section
  recommendation / action / proof section
```

In v0.8, business content differences must not create new public card components.

## Target v0.8 card grammar

| Target component | Responsibility |
|---|---|
| Card | Generic tokenized surface with optional title/description/actions. |
| MetricBlock | Metric value with helper, trend and trust metadata. |
| DecisionBlock | Signal, decision, rationale, recommendation and evidence summary. |
| EvidenceBlock | Source, source scope, freshness, validation, proof readiness and supporting evidence. |
| ActionBlock | Action title, owner, due date, priority, status and context. |
| StatusSummary | Compact structured summary with badges and key-value items. |

## Duplication groups

### Group A — Generic card surfaces

Components:

```txt
Card
AlertCard
RecommendationCard
ActionCard
CustomerStatusCard
ConnectivityCoverageCard
ValueProofCard
AssetIntelligenceSummary
ServiceRiskSummary
RecommendationReviewPanel
```

Current problem:

Many of these components use the same surface grammar but encode different business concepts.

v0.8 direction:

```txt
Card remains the only generic surface.
Business-specific card names must be replaced by DecisionBlock, EvidenceBlock, ActionBlock, StatusSummary or screen-specific patterns.
```

### Group B — Metrics

Components:

```txt
MetricCard
CompactMetric
MetricStrip
MetricGrid
```

Current problem:

Metric display exists as both component-level and decision-level concepts.

v0.8 direction:

```txt
Keep one metric display primitive and one metric grouping primitive.
Avoid dashboard-like metric proliferation in Installed Base.
Use compact metrics only where the screen contract allows them.
```

Recommended target:

```txt
MetricBlock
MetricGroup
```

### Group C — Summary cards

Components:

```txt
StatusSummary
CustomerStatusCard
CustomerReviewReadinessCard
RenewalRiskSummary
```

Current problem:

Several components are simply typed wrappers around key-value summaries.

v0.8 direction:

```txt
StatusSummary becomes the generic summary block.
Business-specific wrappers become internal presets, example data mapping, or are deleted.
```

### Group D — Decision and evidence cards

Components:

```txt
AlertCard
RecommendationCard
ValueProofCard
ServiceRiskSummary
AssetIntelligenceSummary
```

Current problem:

These components mix signal, interpretation, evidence, recommendation and action responsibilities, but with slightly different names.

v0.8 direction:

```txt
DecisionBlock handles signal + decision + recommendation.
EvidenceBlock handles source/proof/freshness/validation.
ActionBlock handles owner/due/priority/actionability.
```

## Keep / merge / internalize / delete recommendations

| Component | Recommendation | Rationale |
|---|---|---|
| Card | Keep | It is the base primitive. It needs visual retokenization. |
| MetricCard | Merge | Same purpose as future MetricBlock; too dashboard-oriented for default Make use. |
| CompactMetric | Merge or keep as MetricBlock compact variant | Useful in Customer Monitoring; should not remain a separate visual grammar. |
| MetricStrip | Merge with MetricGrid | One grouping primitive is enough. |
| MetricGrid | Merge with MetricStrip | Duplicate grouping responsibility. |
| AlertCard | Merge | It is a DecisionBlock with severity and evidence. |
| RecommendationCard | Merge | It is a DecisionBlock with recommendation and evidence. |
| ActionCard | Merge | It is an ActionBlock card variant. |
| CustomerStatusCard | Merge | It is a status/context summary with customer-specific labels. |
| ConnectivityCoverageCard | Merge | It combines metrics and key-values; should be composed from MetricBlock + StatusSummary. |
| CustomerReviewReadinessCard | Internalize/delete | Thin wrapper over StatusSummary and not active golden-critical. |
| RenewalRiskSummary | Internalize/merge | Thin wrapper over StatusSummary. |
| ValueProofCard | Merge | It is an EvidenceBlock/ProofBlock pattern. |
| AssetIntelligenceSummary | Merge | It is a status summary plus decision interpretation. |
| ServiceRiskSummary | Merge | It is a decision/risk summary. |
| RecommendationReviewPanel | Keep as generic ReviewPanel or merge | Its `card/section/drawer` modes may be useful, but it should not stay recommendation-specific. |

## Functional preservation requirements

Card rationalization must preserve:

- title and description;
- badges/tags;
- source scope;
- source strength;
- freshness;
- validation status;
- proof readiness;
- expected outcome;
- evidence summary;
- recommendation text;
- action owner/due/priority;
- selected and review states when used inside lists/panels.

## v0.8 implementation implication

Sprint 2 and Sprint 3 should define the visual tokens and semantic primitives first.

Sprint 4 should then replace the current card proliferation with a small set of generic blocks. Golden examples must be refactored only after the new blocks exist, so functionality is preserved.
