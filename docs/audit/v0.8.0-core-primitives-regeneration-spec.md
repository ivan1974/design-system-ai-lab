# v0.8.0 Core Primitives Regeneration Spec

## Status

Preparation spec for the v0.8 core-primitives regeneration layer.

This document is not an implementation patch. It defines what must be regenerated and what must stay stable before touching component code.

## Source of truth

Read first:

```txt
contracts/v0.8-target-surface.contract.json
contracts/v0.8-regeneration-plan.contract.json
contracts/tokens.contract.json
guidelines/reference/architecture/v0.8-target-surface.md
guidelines/reference/architecture/v0.8-regeneration-plan.md

```

## Foundation lock

Before regenerating core primitives, the following files must remain stable:

```txt
src/design-system/foundations/tokens.css
src/design-system/styles.css
contracts/tokens.contract.json
contracts/visual-rules.contract.json
guidelines/tokens.md
guidelines/styles.md
guidelines/runtime/visual-rules.md
```

Do not create:

```txt
src/design-system/tokens.css
```

Any core primitive change must consume existing `--ec-*` tokens. It must not introduce a new token source.

## Target core primitives

The v0.8 core primitive layer includes:

```txt
Button
Card
Table
Tabs
Divider
Dialog
Popover
Tooltip
Checkbox
Field
Input
Label
Select
Textarea
```

These are low-level UI primitives. They must not encode business meaning.

## Role of core primitives

Core primitives should provide:

- tokenized visual consistency;
- accessible low-level interaction;
- compact B2B density;
- shadcn-compatible internal patterns where useful;
- stable props documented in `contracts/props.contract.json` when values affect generation.

Core primitives should not provide:

- business labels;
- source strength semantics;
- proof readiness semantics;
- decision ownership;
- app-specific layout;
- one-off screen behavior.

## Visual rules

Core primitives must follow:

```txt
white-first surfaces
borders before shadows
compact typography
compact controls
neutral input background
single radius scale
meaningful accent usage only
```

They must use tokens from:

```txt
src/design-system/foundations/tokens.css
```

## Primitive-specific targets

### Button

Target:

```txt
variants: primary | secondary | ghost | danger | outline
sizes: sm | md | lg | icon
```

Rules:

- primary means main action;
- danger means destructive or irreversible action;
- ghost and outline remain secondary;
- no business-specific action variant.

### Card

Target:

```txt
base contained surface
compact density support
neutral tone by default
```

Rules:

- `Card` is a primitive, not a business component;
- do not create a new public card for a content label;
- use target blocks for metrics, decisions, evidence and actions after regeneration.

### Table

Target:

```txt
compact operational comparison
stable row and column structure
```

Rules:

- inventories and queues prefer rows/tables over card grids;
- row density must stay compact;
- explanation and evidence move to panels or secondary structures.

### Tabs

Target:

```txt
variants: underline | contained
sizes: sm | md
```

Rules:

- use tabs for stable content sections;
- do not use tabs as filters;
- do not create local tab systems.

### Dialog, Popover and Tooltip

Target:

```txt
package-owned overlay primitives
```

Rules:

- no local overlay implementation;
- no local tooltip implementation;
- use for interaction support, not business-specific panels.

### Forms

Target:

```txt
Checkbox
Field
Input
Label
Select
Textarea
```

Rules:

- compact density;
- controlled values only when generation choices change;
- no local form control styling;
- no raw form controls in generated examples.

They are either low-level support, transitional utilities, or candidates for later rationalization.

## Regeneration approach

Do not rewrite all primitives blindly.

For each target primitive:

```txt
1. inspect current source
2. confirm token usage
3. remove hardcoded visual values when safe
4. align props with contracts
5. update guideline only if the component is regenerated
6. update story coverage when release-ready
7. keep npm run check green after each slice
```

## Exit criteria

Core primitives regeneration is complete when:

```txt
foundations lock passes
all target core primitives consume --ec-* token grammar
no target primitive creates a local token system
no target primitive encodes business semantics
props contract matches generation-relevant values
runtime component selection points to target primitives
storybook coverage is restored to release-relevant for regenerated primitives
npm run check passes
```
