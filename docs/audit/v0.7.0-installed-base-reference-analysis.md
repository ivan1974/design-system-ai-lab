# v0.7.0 Installed Base reference analysis

## Status

Sprint 1 contract-first foundation.

This document turns the Installed Base Intelligence prototype specification into a Design System AI Lab source of truth for v0.7.0.

The goal is not to reproduce a local prototype. The goal is to make the Make Kit capable of generating the Installed Base Intelligence screen using only public design-system components, public design-system patterns, design-system tokens, runtime rules, reference guidelines, screen contracts and domain contracts.

## Reference prototype source

The reference prototype is the Installed Base Intelligence prototype architecture specification.

It is a closed specification. Compliance requires all mandatory elements to be present. The normative terms MUST, MUST NOT, ALWAYS, NEVER, EXACTLY, ONLY and REQUIRED are binding.

Any behavior, content, interaction, navigation element, filter, view, panel, tab, field, column, card, section, action or visual element not explicitly defined in the specification is not allowed.

## Required primary layers

The screen MUST contain exactly five primary layers.

The order MUST be preserved:

1. Main Navigation
2. Page Header
3. View & Filter Bar
4. Installed Base List
5. Asset Detail Panel

No additional primary layers are allowed.

## Mandatory layer responsibilities

### Main Navigation

Global application navigation only.

It contains exactly:

1. Schneider Electric Logo
2. Product Name
3. Global Search
4. Help
5. User Menu

It MUST NOT contain site, customer, campus, portfolio, asset, filter, view selector, KPI, marketing navigation or public website navigation content.

### Page Header

Installed base context only.

It contains exactly:

1. Context Pill
2. Site Context
3. Portfolio Title
4. Portfolio Scope Summary
5. Asset Intelligence

It MUST NOT contain filters, asset information, tab navigation, actions, recommendations or health information.

### View & Filter Bar

View mode and quick filters only.

It contains exactly two zones:

1. Left Zone
2. Right Zone

Left Zone contains exactly three views:

1. List
2. Geography
3. Electrical

Right Zone contains exactly four filter triggers:

1. Location
2. Asset Type
3. Contract Type
4. All Filters

No additional views or filters are allowed.

### Installed Base List

Grouped asset portfolio only.

The grouping hierarchy is exactly:

```txt
Building → Electrical Area → Room → Asset
```

Assets with active alert or connectivity issue appear first under:

```txt
Attention required
```

The list contains exactly seven columns:

1. Asset
2. Type
3. Location
4. Coverage
5. Health
6. Status
7. Action

### Asset Detail Panel

Selected asset analysis only.

It contains exactly four sections:

1. Panel Header
2. Tab Navigation
3. Tab Content
4. Action Area

It contains exactly six tabs:

1. Overview
2. Health
3. Intelligence
4. Passport
5. History
6. Documents

The Action Area contains only:

1. Schedule Service
2. Download Report
3. Contact Expert

## Known inconsistency

The reference specification states:

```txt
Page Header MUST contain EXACTLY four elements
```

but lists five elements:

1. Context Pill
2. Site Context
3. Portfolio Title
4. Portfolio Scope Summary
5. Asset Intelligence

## DS decision

The v0.7.0 screen contract resolves the inconsistency by defining the Page Header as containing exactly five elements.

Asset Intelligence is required because it carries the screen-specific promise:

```txt
Component hierarchy + intelligence enabled
```

Removing it would weaken the Installed Base Intelligence screen contract.

## Third-party asset rule

A third-party asset MUST always display:

| Field | Value |
| --- | --- |
| Connectivity | Unknown |
| Coverage | No Coverage |
| Health | Unknown |
| Status | No record |

No exceptions are allowed.

## Mandatory interaction rules

The screen contract must preserve these interactions:

- selecting an asset row opens the Asset Detail Panel;
- clicking the overlay closes the panel;
- clicking the close icon closes the panel;
- pressing Escape closes the panel;
- background content does not scroll when a panel is open;
- the Installed Base List does not scroll when the Asset Detail Panel is open;
- only panel content scrolls;
- the Action Area remains visible while panel content scrolls;
- multiple values can be selected in All Filters Panel;
- Clear All clears selected filters.

## Mandatory context preservation

The following context must remain preserved during navigation:

- Active View
- Active Filters
- Selected Asset
- Selected Tab

## Fact / interpretation separation

The Health and Intelligence tabs must remain separated.

| Area | Required content |
| --- | --- |
| Health | facts, measurements, observations, current condition, telemetry, alert facts, freshness, source |
| Intelligence | diagnosis, interpretation, benchmarking, recommendations, expected outcome, proof readiness, source strength, validation, action plan |

Content MUST NOT be duplicated between Health and Intelligence.

## Progressive decision disclosure opportunity

The reference prototype is strict. v0.7.0 must preserve that strictness while improving cognitive hierarchy.

The operating principle is:

```txt
Generated decision screens must reveal decision-critical information first, then expose evidence and explanation progressively.
```

This means:

1. Signal first: what requires attention?
2. Decision second: what should be done?
3. Evidence third: why should the user trust it?

## Visual system implications

The screen requires an operational intelligence visual system:

- dense rows over repeated cards;
- table-like alignment for inventory data;
- white-first surfaces;
- subtle borders;
- controlled list width;
- compact tags for categories;
- pills for health and state values;
- icon + label for operational statuses;
- Schneider green only for selected, active, primary or positive state;
- no hero dashboards;
- no card grids replacing the inventory table;
- no decorative gradients;
- no local colored panels;
- panel width between 360px and 500px.
