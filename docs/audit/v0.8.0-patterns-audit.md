# v0.8.0 Patterns Audit

## Status

Sprint 1 audit for v0.8.0.

This document audits `src/design-system/patterns/`. The goal is to decide which patterns are reusable DS compositions, which are screen-contract patterns, and which are application-specific wrappers that should be internalized or deleted.

## Pattern layer principle for v0.8

A pattern belongs in the public DS only if it provides reusable structure for Make generation.

A pattern should not remain public when it only:

- maps one business object to a generic card;
- contains demo copy or mock-specific assumptions;
- wraps a generic component without adding reusable layout or interaction responsibility;
- belongs to application workflow rather than DS composition.

## Screen-contract patterns

These patterns are justified because Installed Base is the contractual screen:

```txt
AllFiltersPanel
AssetDetailAnalysisPanel
AssetInventoryRow
InstalledBaseHeader
InstalledBaseList
InstalledBaseViewFilterBar
```

They should remain public, but their internals must follow v0.8 rationalization:

- no HealthPill / CoverageTag / DppTag / ConnectivityLabel / StatusWithIcon as public semantic dependencies;
- use semantic primitives plus contract-controlled mappings;
- use global tokens from the prototype visual grammar;
- no local decision cards inside panel content;
- no hardcoded colors or arbitrary visual values.

## Customer Monitoring patterns

Customer Monitoring currently depends on:

```txt
CustomerStatusCard
ConnectivityCoverageCard
AssetQueueRow
AlertCard
ActionRow
```

v0.8 direction:

- CustomerStatusCard becomes a generic summary/context block, not a customer-specific public card.
- ConnectivityCoverageCard becomes a composition of MetricBlock + StatusSummary / CoverageSummary.
- AssetQueueRow becomes a ReviewQueueRow preset or internal wrapper.
- AlertCard becomes DecisionBlock.
- ActionRow remains as compact operational row.

Customer Monitoring must keep its functional capability, but it should no longer require its own mini pattern set.

## Renewal and recommendation patterns

Renewal Risk and Asset Recommendation currently depend on business-specific summary cards and review cards:

```txt
RenewalRiskSummary
ValueProofCard
AssetIntelligenceSummary
ServiceRiskSummary
RecommendationReviewPanel
RecommendationCard
RiskQueueRow
RecommendationQueueRow
```

v0.8 direction:

- RenewalRiskSummary becomes StatusSummary data mapping.
- ValueProofCard becomes EvidenceBlock / ProofBlock.
- AssetIntelligenceSummary becomes StatusSummary + DecisionBlock or generic IntelligenceSummary.
- ServiceRiskSummary becomes DecisionBlock or StatusSummary.
- RecommendationReviewPanel becomes generic ReviewPanel if its review-context structure remains useful.
- RiskQueueRow and RecommendationQueueRow become ReviewQueueRow presets/internal wrappers.

## DS vs app-specific boundary

Patterns should not own:

- data fetching;
- backend filtering;
- persistence;
- routing;
- authorization;
- analytics;
- demo-specific customer data;
- one-off copy blocks;
- one app's state transitions.

Patterns may own:

- reusable layout;
- accessible interaction structure;
- panel behavior;
- list/detail composition;
- controlled visual hierarchy;
- contract-enforced screen sections;
- reusable decision/evidence/action layout.

| Keep or genericize | RecommendationReviewPanel, ComponentHierarchy, CreateActionDialog |
| Merge | CustomerStatusCard, ConnectivityCoverageCard, ValueProofCard, AssetIntelligenceSummary, ServiceRiskSummary |
| Internalize | AssetQueueRow, CustomerQueueRow, RecommendationQueueRow, RiskQueueRow, RenewalRiskSummary, CustomerReviewReadinessCard |
| Delete candidate | Any pattern not used by active examples, not referenced by contracts, and not needed by Customer Monitoring or Installed Base generation |

## Sprint 1 conclusion

The patterns folder contains both legitimate screen-contract patterns and multiple business wrappers over generic structures.

v0.8 should keep the screen-contract patterns and reduce the business-specific wrappers into generic, contract-controlled compositions.
