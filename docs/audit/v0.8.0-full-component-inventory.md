# v0.8.0 Full Component Inventory Audit

## Status

Sprint 1 audit for v0.8.0.

This file inventories the active Make-facing design-system surface. The goal is not to optimize one screen. The goal is to ensure that every component used by active examples is audited before v0.8 refactor work starts.

## Audit basis

The inventory is based on:

- public exports in `src/design-system/index.ts`;
- active golden examples in `guidelines/examples/golden/`;
- screen-contract stories and golden stories;
- implementation spot checks for cards, decision components, semantic display components and patterns;
- the v0.8 requirement that Installed Base and Customer Monitoring must share the same visual grammar without functional regression.

## Decision vocabulary

| Decision | Meaning |
|---|---|
| Keep | Keep as public DS surface, then retokenize and align visually. |
| Merge | Merge into a smaller primitive or pattern family. Do not keep as a separate public concept unless a structural difference is proven. |
| Internalize | Keep implementation if useful, but remove from public Make-facing surface or treat as screen-specific internals. |
| Delete | Remove if unused, legacy or fully replaced. |
| Review | Insufficient evidence for final decision; must be resolved before v0.8 release. |

## High-level finding

The public package surface is currently broad. It exposes many primitives, composition components, decision components and business patterns. Several public components encode business labels while sharing the same visual structure. This is especially visible in pills, tags, queue rows, summary cards and review cards.

The v0.8 target is a smaller public grammar:

```txt
Core primitives: Button, Card, Table, Tabs, Form controls, SidePanel, ListContainer
Semantic display: SemanticTag, SemanticPill, StatusIndicator, MetaLabel
Decision structures: MetricBlock, DecisionBlock, EvidenceBlock, ActionBlock, StatusSummary
Screen patterns: Installed Base patterns + Customer Monitoring generation pattern
```

## Immediate Sprint 1 conclusions

1. The active golden examples are not all using the same visual grammar today.
2. Customer Monitoring relies on several business-pattern cards; Installed Base relies on screen-specific patterns.
3. Many decision display components are thin mappings from business values to identical pill/tag/badge structures.
4. Several queue-row components are wrappers around the same `ReviewQueueRow` structure.
5. Several pattern cards use the same `Card + SectionStack + KeyValueList + tags` structure.
6. Storybook coverage is too often justified through integrated stories rather than explicit component stories, which is acceptable for v0.7 but not sufficient for v0.8.

## Sprint 1 exit status

No active example component is intentionally out of audit scope.

The next sprint can start from this classification and create the target v0.8 component grammar before code refactoring begins.
