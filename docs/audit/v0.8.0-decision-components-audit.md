# v0.8.0 Decision Components Audit

## Status

Sprint 1 audit for v0.8.0.

This document audits `src/design-system/decision/`. The goal is to separate true reusable decision structures from business-label wrappers and visual variants.

## Decision folder principle for v0.8

The `decision` folder should contain cognitive and operational decision structures, not business-specific cards renamed by use case.

A component belongs in `decision` only if it helps structure one of these responsibilities:

- signal visibility;
- decision guidance;
- evidence and trust;
- actionability;
- review readiness;
- queue prioritization;
- status or proof summary.

A component should not remain in `decision` if it only maps a business value to a pill, tag or color.

## Components that are true decision structures

### Keep or evolve

```txt
ReviewQueueRow
ActionRow
EvidenceRow
StatusSummary
```

These have reusable structure and support multiple examples.

### Replace with new decision structures

```txt
AlertCard
RecommendationCard
ActionCard
```

These should become:

```txt
DecisionBlock
ActionBlock
EvidenceBlock
```

### Merge into semantic display primitives

```txt
HealthPill
CoverageTag
DppTag
ConnectivityLabel
PriorityPill
SourceStrengthPill
StatusPill
StatusWithIcon
```

These are mostly mappings from controlled values to labels, tones and icons. Their meaning should move to contracts and mapping utilities.

Target primitives:

```txt
SemanticTag
SemanticPill
StatusIndicator
MetaLabel
```

## Required contract migration

The following mappings should no longer live mainly in public component names:

- health value to tone;
- coverage value to label/tone;
- DPP value to label/tone;
- connectivity value to indicator/tone;
- operational status to icon/tone/label;
- priority to tone/label;
- source strength to tone/label.

They should be represented in:

```txt
contracts/semantic-display.contract.json
contracts/props.contract.json
contracts/installed-base-domain.contract.json
contracts/component-registry.contract.json
```

## Risks to avoid

1. Replacing specialized components with generic primitives without contracts would make Make too permissive.
2. Keeping specialized components would preserve v0.7 component proliferation.
3. Moving all decision content to generic Card would lose cognitive structure.

v0.8 must therefore combine:

```txt
small semantic display primitives
+ controlled mappings
+ reusable decision blocks
+ strict screen contracts
```

## Sprint 1 conclusion

The decision folder should be significantly reduced.

The target is not fewer capabilities. The target is fewer public component types with stronger contracts and more consistent rendering.
