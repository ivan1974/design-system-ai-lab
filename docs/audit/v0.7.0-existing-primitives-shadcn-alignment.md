# v0.7.0 existing primitives shadcn alignment audit

## Status

Sprint 3.5 radical primitive refoundation artifact.

## Decision

The DS primitive layer will be aligned with shadcn-compatible React, Tailwind and Radix patterns where useful.

The public generation surface remains `design-system-ai-lab`.

Generated examples must never import shadcn, Radix or local `components/ui` primitives directly.

| Badge | `tone` | partially aligned | adapt next; keep tone semantics, add variant-like behavior only if contract needs it |

## Radical refoundation principle

Radical does not mean replacing every file immediately.

It means every primitive has an explicit status:

```txt
aligned
partially aligned
acceptable for now
use with care
optional later migration
```

The DS should not carry unknown primitive debt.

## What is intentionally not migrated now

Do not migrate business components directly:

```txt
HealthPill
CoverageTag
DppTag
StatusWithIcon
InstalledBaseHeader
InstalledBaseList
AssetInventoryRow
AssetDetailAnalysisPanel
```

They must remain business wrappers controlled by contracts.

## Sprint 4 readiness impact

Sprint 4 can use:

```txt
Button
Checkbox
ScrollArea
Popover
SidePanel
Tabs
Input
Select
```

without creating local `components/ui` primitives.

## Remaining primitive work after Sprint 4 starts

Only revisit primitives when a screen component exposes a concrete limitation:

- migrate `Select` internals to Radix if native select blocks advanced filters;
- migrate `Checkbox` internals to Radix if accessibility needs exceed native behavior;
- migrate `Popover` internals to Radix only with a deliberate lockfile update;
- add variant support to `Badge` when a contract needs it.

## Guardrail

A primitive implementation change is allowed only when the public DS API, props contract, registry entry, guideline and story coverage stay aligned.
