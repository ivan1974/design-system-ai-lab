# v0.8.0 Example Usage Map

## Status

Sprint 1 audit for v0.8.0.

This document maps which public design-system components are used by active Make-facing examples. Its purpose is to prevent v0.8 from optimizing only Installed Base while leaving Customer Monitoring, Renewal Risk or Asset Recommendation on an older visual grammar.

## Active golden examples

The active examples are:

```txt
guidelines/examples/golden/customer-monitoring.App.tsx
guidelines/examples/golden/installed-base-intelligence.App.tsx
guidelines/examples/golden/renewal-risk-review.App.tsx
guidelines/examples/golden/asset-recommendation-review.App.tsx
```

Installed Base Intelligence is the strict contractual screen.

Customer Monitoring is the second v0.8 generation proof.

Renewal Risk Review and Asset Recommendation Review remain active examples and must not keep a legacy component grammar.

## Shared structural layer

The non-Installed examples already share a useful screen skeleton:

```txt
WorkspaceShell
  PageHeading
  FilterBar
  MasterDetailLayout
    ListContainer + queue rows
    WorkspaceDetailPanel + Tabs + SectionStack
  ActionRow follow-through section
```

This skeleton should be preserved in v0.8 because it supports Customer Monitoring, Renewal Risk and Asset Recommendation without rebuilding an app-specific layout.

## Differences to rationalize

### Header and navigation

| Current state | v0.8 direction |
|---|---|
| Customer Monitoring, Renewal and Asset Recommendation use PageHeading. | Keep PageHeading as generic page-level heading. |
| Installed Base uses MainNavigation plus InstalledBaseHeader. | Keep because the screen contract requires global nav and dedicated product header. |
| Breadcrumbs appear only in Customer Monitoring. | Review whether breadcrumbs are core DS or example-level context. |

### Filtering and controls

| Current state | v0.8 direction |
|---|---|
| Customer Monitoring, Renewal and Asset Recommendation use FilterBar. | Keep FilterBar as generic workspace filter/summary bar. |
| Installed Base uses InstalledBaseViewFilterBar. | Keep as screen-contract pattern, but align its visuals with FilterBar and the prototype. |
| AllFiltersPanel exists only for Installed Base. | Keep as contractual side-panel filter pattern. |

### List and queue rows

## Functional preservation checklist

The rationalization must not remove any of the following capabilities:

### Customer Monitoring

- customer context summary;
- monitoring scope filters;
- coverage and visibility limits;
- metric strip for monitoring signals;
- asset queue with selected row state;
- detail panel with tabs;
- coverage detail;
- priority alert/recommendation content;
- assigned next actions.

### Installed Base Intelligence

- global navigation;
- Installed Base header;
- exact view/filter bar;
- all filters panel;
- grouped inventory table;
- attention-required group;
- exact seven-column structure;
- six asset states;
- third-party mandatory value normalization;
- asset detail side panel;
- six tabs;
- sticky actions;
- Health / Intelligence separation.

### Renewal Risk Review

- renewal summary;
- blocker queue;
- value proof content;
- detail panel;
- recommendation review;
- mitigation actions.

### Asset Recommendation Review

- asset intelligence context;
- recommendation queue;
- service risk context;
- recommendation detail panel;
- evidence rows;
- follow-through actions.

## Output decision

The active examples are functionally valid but visually and structurally too fragmented.

v0.8 should not remove their capabilities. It should replace the fragmented component surface with a shared grammar:

```txt
Screen shell + FilterBar + ListContainer/ReviewQueueRow + DetailPanel/SidePanel + Semantic display + Decision/Evidence/Action blocks
```

Installed Base keeps its stricter screen-specific patterns, but those patterns must use the same semantic display primitives and visual tokens as Customer Monitoring.
