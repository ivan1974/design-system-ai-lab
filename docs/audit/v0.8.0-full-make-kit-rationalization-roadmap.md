# v0.8.0 Full Design System Rationalization and Visual Alignment Roadmap

## Status

Corrected target roadmap for v0.8.0.

v0.8.0 is not an Installed Base refactor with a few rationalized primitives around it. It is a systemic rationalization of the whole Make Kit surface used by examples, golden examples, stories, contracts, prompts and benchmarks.

## Final formulation

v0.8.0 is a full Make Kit rationalization release.

Installed Base remains the strict contractual screen.

Customer Monitoring becomes the second generation proof.

The entire active design system surface is restyled, simplified, aligned and tested.

## Guiding principle

Installed Base is the contractual screen, but the whole Make Kit surface must be rationalized, restyled and aligned.

No component can remain in the Make Kit surface only because it already exists.

A component must justify at least one of these:

- a distinct visual structure;
- a distinct interaction responsibility;
- a reusable composition role;
- a strict screen-contract responsibility.

Same visual structure means same component.

Different business content does not justify a new public component.

Business differences must be handled by content, props, mappings, slots and contracts.

## Strategic pivot after Sprint 3

The previous incremental refactor created too much friction because legacy code, legacy guidelines and legacy tests still preserved the old component surface.

From this point, the v0.8 approach becomes:

```txt
Target surface first.
Guidelines and contracts second.
Code regeneration third.
Legacy deletion last.
```

Do not continue patching every legacy component one by one unless it blocks the target reset.

The source of truth for v0.8 is the target surface contract, not the accumulated legacy implementation.

## Card rationalization rule

A card is not a component type because its content is different.

A card becomes a separate component only if its structure, interaction behavior or reusable composition responsibility is different.

If several cards have the same density, header, body, footer, action area and information hierarchy, they must be merged.

Target grammar:

```txt
Card
MetricBlock
DecisionBlock
EvidenceBlock
ActionBlock
StatusSummary
```

Likely merges:

```txt
MetricCard variants -> MetricBlock
AlertCard -> DecisionBlock
RecommendationCard -> DecisionBlock
ActionCard -> ActionBlock
EvidenceCard -> EvidenceBlock
InsightCard -> DecisionBlock or Card
RiskCard -> DecisionBlock or StatusSummary
CustomerCard -> StatusSummary or Card
AssetCard -> StatusSummary or Card
```

The difference between use cases must come from props, slots, content and contracts, not from unnecessary specialized components.

## DS vs application boundary

The DS owns:

- visual primitives;
- semantic display primitives;
- reusable decision structures;
- reusable composition patterns;
- screen contracts;
- design tokens;
- allowed values and mappings;
- generation rules;
- repair rules;
- benchmark expectations.

Applications own:

- data fetching;
- backend filtering;
- routing;
- authorization;
- persistence;
- real customer data;
- real asset lifecycle logic;
- analytics events;
- application-specific flows.

If a component only encodes one screen's mock data, one app's copy, or one use-case wrapper without reusable DS value, it must be internalized or deleted.

## Alignment rule

No component, token, pattern or screen rule is valid until it is:

- documented in guidelines;
- represented in a contract when relevant;
- implemented in code;
- exported if public;
- covered by Storybook or explicitly justified;
- covered by tests;
- reflected in golden examples if used;
- reflected in prompts if used by Make;
- reflected in benchmarks and repair prompts when relevant.

## v0.8 global exit criteria

v0.8.0 succeeds only if:

1. all components used by active examples have been audited;
2. all cards have been rationalized;
3. all decision components have been rationalized;
4. all active patterns have been rationalized;
5. visually identical components have been merged;
6. business role differences are handled by content, props, mappings or contracts;
7. the prototype visual system is applied to the whole DS;
8. tokens replace hardcoded styles across the active surface;
9. Installed Base remains the strict contractual screen;
10. Customer Monitoring becomes the second generation proof;
11. active golden examples share the same visual grammar;
12. prompts exist for Installed Base and Customer Monitoring;
13. prompts generate without local components;
14. benchmarks penalize component proliferation;
15. adversarial prompts cover redundant cards and semantic variants;
16. app-specific components are deleted, internalized or moved out of DS;
17. legacy components are deleted;
18. guidelines align with code;
19. contracts align with code;
20. stories align with public exports;
21. tests cover the public surface;
22. benchmarks reflect the v0.8 strategy;
23. repair prompts reflect v0.8 failure modes;
24. README and CHANGELOG reflect the v0.8 surface;
25. `npm run check` passes;
26. `npm run build-storybook` passes;
27. `npm pack --dry-run` shows a clean package surface.

## Final reminder

The prototype defines the visual quality bar for the whole DS, not only for Installed Base.

The roadmap is not about preserving the old surface.

The roadmap is about defining the v0.8 target surface, regenerating the DS from that target, then deleting legacy.
