# v0.8.0 Full Make Kit Rationalization Roadmap

## Status

Target roadmap for v0.8.0.

v0.8.0 is not a screen-specific refactor. It is a full Make Kit rationalization and visual alignment release.

Installed Base Intelligence remains the strict contractual screen. Customer Monitoring becomes the second generation proof. The entire active design system surface used by Make-facing examples must be rationalized, restyled, aligned and tested.

## Strategic goal

Move from v0.7.0-alpha.0 Installed Base screen-contract proof to a smaller, stricter and more visually consistent Make Kit that can generate multiple product screens without local components, visual drift or component proliferation.

The prototype provided for v0.8 defines the visual quality bar for the whole design system, not only for Installed Base.

## Non-negotiable principles

1. The prototype defines the visual grammar for the whole active DS surface.
2. Installed Base is the contractual screen, but the DS rationalization is global.
3. Customer Monitoring must be generable with the same visual quality and component grammar.
4. No component can remain public only because it already exists.
5. Same visual structure means same component.
6. Different business content does not justify a new component.
7. Business meaning is controlled by contracts, props and allowed values, not by one component per label.
8. Components used by active examples must be audited, rationalized, documented, contracted, storied and tested.
9. Application-specific logic must not remain in the design system surface.
10. Legacy artifacts must be deleted or moved outside the active Make surface.

## Target capability

After v0.8.0, one prompt must be able to generate:

- Installed Base Intelligence, with strict screen-contract compliance.
- Customer Monitoring, with the same visual system, density, token usage and rationalized component grammar.

The refactor must not degrade functionality or interaction capability. It should preserve or improve:

- filtering;
- table/list density;
- panel interactions;
- progressive decision disclosure;
- evidence and trust display;
- actionability;
- screen-level context preservation;
- benchmark repairability.

## Core architectural decision

v0.7 exposed several business-specific display components for Installed Base. v0.8 must rationalize this.

The screen contract controls meaning. The design system controls visual grammar.

Do not keep one public component per business label when the rendering structure is identical.

### Replace business-specific display variants with semantic primitives

Preferred v0.8 semantic primitives:

```txt
SemanticTag
SemanticPill
StatusIndicator
MetaLabel
```

Specialized components that should be deleted, merged or internalized unless a distinct structure is justified:

```txt
HealthPill
CoverageTag
DppTag
ConnectivityLabel
StatusWithIcon
PriorityLabel
ProofStatus
SourceStrength
```

The same principle applies to cards and decision components.

## Card rationalization target

All cards across components, decision and patterns must be audited.

A card is not a component type because its content is different. A card becomes a separate component only if its visual structure, interaction behavior or reusable composition responsibility is different.

Target grammar:

```txt
Card
MetricBlock
DecisionBlock
EvidenceBlock
ActionBlock
StatusSummary
```

Likely merges:

```txt
AlertCard -> Card or DecisionBlock
RecommendationCard -> DecisionBlock
ActionCard -> ActionBlock
EvidenceCard -> EvidenceBlock
InsightCard -> DecisionBlock or Card
MetricCard variants -> MetricBlock
RiskCard -> DecisionBlock or StatusSummary
```

The final merge list must be based on the real inventory audit.

## DS vs application boundary

The DS owns:

- visual primitives;
- semantic display primitives;
- reusable decision structures;
- reusable composition patterns;
- screen contracts;
- design tokens;
- allowed values and mappings;
- generation rules;
- repair rules;
- benchmark expectations.

Applications own:

- data fetching;
- backend filtering;
- routing;
- authorization;
- persistence;
- real customer data;
- real asset lifecycle logic;
- analytics events;
- application-specific flows.

If a component only encodes one app's data or screen copy, it is not a DS component.

## Alignment rule

No component, token, pattern or screen rule is valid until it is:

- documented in guidelines;
- represented in a contract when relevant;
- implemented in code;
- exported if public;
- covered by Storybook;
- covered by tests;
- reflected in golden examples if used;
- reflected in prompts if used by Make;
- reflected in benchmarks and repair prompts when relevant.
