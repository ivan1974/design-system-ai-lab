# v0.7.0 Sprint 3.5 radical primitive summary

## Status

Sprint 3.5 radical primitive foundation is substantially complete.

## Core decision

```txt
We align implementation with shadcn-compatible patterns.
We align generation with contracts.
```

## What changed

### Governance

Created the primitive architecture contract, runtime strategy and test coverage for the new boundary:

```txt
shadcn-compatible internally
design-system-ai-lab externally
```

Generated examples must not import:

```txt
@radix-ui/*
components/ui
src/design-system/internal
src/design-system/primitives
design-system-ai-lab/src
design-system-ai-lab/dist
```

### New primitives

Added DS public primitives needed by Sprint 4, Sprint 5 and Sprint 6:

```txt
Table
ScrollArea
Checkbox
Tooltip
Popover
```

### Existing primitives

Audited existing primitives and gave each one an explicit alignment status.

`Button` was adapted immediately with shadcn-like `outline` and `icon` values while keeping the DS public API.

Other primitives were classified for later adaptation only when a concrete screen need appears.

## Intentional non-migration

Business components remain wrappers controlled by contracts:

```txt
HealthPill
CoverageTag
DppTag
StatusWithIcon
InstalledBaseHeader
InstalledBaseList
AssetInventoryRow
AssetDetailAnalysisPanel
```

They should not expose primitive internals.

## Known follow-up work

- Badge variant support can be added when needed by a screen contract.
- Select can move from native select to Radix only if advanced filter UX requires it.
- Checkbox can move to Radix only if native behavior is insufficient.
- Popover can move to Radix only with an intentional lockfile update.
- Internal primitive folder can be introduced later if code reuse pressure appears.

## Sprint 4 readiness

Sprint 4 may now build:

```txt
MainNavigation
InstalledBaseHeader
InstalledBaseViewFilterBar
AllFiltersPanel
```

using DS public primitives only.

No local `components/ui` primitives should be created.
